# ТЗ / PRD: Одностраничный лендинг InsightStream (RU, GitHub Pages)

## 1. Назначение

Документ задаёт исчерпывающее ТЗ на одностраничный лендинг продукта InsightStream:
- структура страницы;
- контент и правила формулировок;
- техстек и папка проекта;
- схема публикации на GitHub Pages;
- схема ссылок на актуальные desktop-сборки из CI/release.

---

## 2. Цель лендинга

Главная цель:
- мотивировать пользователя скачать демо desktop-приложения.

Вторичная цель:
- объяснить, как после демо получить/купить ключ и продолжить использование.

Лендинг не должен быть документационным порталом.
Нужна только минимальная пользовательская справка в рамках одной страницы.

---

## 3. Продуктовая рамка

## 3.1 Позиционирование

InsightStream — RPA-агент для повседневной работы на компьютере:
- порядок в файлах и папках;
- поиск нужной информации в файловой системе;
- помощь с документами;
- помощь с базовой настройкой компьютера.

## 3.2 Модель доступа

- Есть бесплатный демо-режим.
- После демо доступ по ключу.

## 3.3 Тариф

- Один тариф: **5000 ₽ / месяц**.
- Пользователь получает ключ и активирует его в приложении.
- Текущий канал получения ключа: запрос на `bavadim@gmail.com`.
- Оплата: по счёту из email-ответа.

## 3.4 Публичные ограничения коммуникации

На лендинге:
- не писать про `$1` демо-кредит;
- не писать про OpenAI, `$20`, токены и внутреннюю экономику;
- писать: «можно попробовать бесплатно»;
- писать: «после демо — ключ 5000 ₽/мес».

## 3.5 Операционные условия (мягкие, для небольшой команды)

- Выдача ключа: best-effort, обычно в день обращения.
- Поддержка: best-effort в рабочие часы по Москве.
- SLA с жёсткими штрафами и гарантиями аптайма на этом этапе не вводится.
- Возвраты/отмена: через поддержку, решения принимаются индивидуально.
- Канал заявок на ключ и обратной связи: `bavadim@gmail.com`.
- Целевой срок после подтверждённой оплаты: ключ отправляется в течение 1 рабочего дня (обычно в тот же день).

---

## 4. Целевая аудитория

- Владельцы и операционные менеджеры SMB.
- Офисные сотрудники (операции, бэк-офис, ассистенты).
- Нетехнические пользователи.

Ключевые возражения:
- сложно;
- небезопасно;
- непонятно, что будет в демо;
- непонятно, стоит ли 5000 ₽/мес.

Лендинг должен закрывать эти возражения в явных блоках.

---

## 5. Формат сайта

Сайт — **одна страница** (`/`) с якорной навигацией.

Секции в порядке сверху вниз:
1. Hero
2. Что умеет
3. Кейсы
4. Скачать демо
5. Тариф и ключ
6. FAQ / Помощь
7. Footer

## 6. Требования к контенту секций

## 6.1 Hero

Обязательные элементы:
- простое H1 без техтерминов;
- подзаголовок «для кого и для каких задач»;
- primary CTA: `Скачать демо`;
- secondary CTA: `Тариф и ключ`;
- короткий блок «что можно делать бесплатно» (без $ и токенов).
- короткая строка под CTA: «Нужен ключ — `bavadim@gmail.com`».

## 6.2 Что умеет

4 карточки:
- файлы/папки;
- поиск;
- документы;
- базовая настройка ПК.

Каждая карточка: «действие -> результат для пользователя».

## 6.3 Кейсы

- 6–8 коротких кейсов.
- Формат: «проблема -> что делает агент -> результат».
- Допускаются оценочные формулировки экономии времени (без гарантий).

## 6.4 Скачать демо

Обязательные элементы:
- кнопки скачивания desktop по платформам;
- системные требования;
- 3 шага запуска;
- заметка: «демо бесплатно, после демо можно активировать ключ».
- ссылка/подсказка рядом: «Запрос ключа: `bavadim@gmail.com`».
- отдельный trust-блок рядом со скачиванием.

Trust-блок (обязательно):
- кто делает продукт (кратко, 1–2 строки);
- как запрашивается ключ и поддержка (email);
- что делать при проблеме установки (контакт + шаги);
- ссылка на upstream Goose для технической прозрачности.

## 6.5 Тариф и ключ

Обязательные элементы:
- цена `5000 ₽/мес`;
- что включено;
- когда нужен ключ;
- как получить ключ (через email);
- как активировать в приложении;
- CTA `Запросить ключ` (mailto: `bavadim@gmail.com`).

Отдельно в секции:
- короткая инструкция: «Нужен ключ — напишите на `bavadim@gmail.com`».
- ожидаемый формат письма: имя, компания (опционально), цель использования.
- прозрачный платежный flow:
  1. пользователь пишет на почту;
  2. в ответ получает счёт;
  3. после оплаты получает ключ;
  4. ключ отправляется в течение 1 рабочего дня.

## 6.6 FAQ / Помощь

Минимальные вопросы:
- что можно делать бесплатно;
- что будет после демо;
- безопасно ли для файлов;
- что делать, если не работает;
- как связаться с поддержкой.

Дополнительно:
- блок «кому подходит / не подходит».
- явный пункт: «Где получить ключ? -> написать на `bavadim@gmail.com`».
- явный пункт: «Как проходит оплата? -> по счёту из email, затем выдача ключа».
- явный пункт: «Что будет после окончания демо? -> в приложении появится окно с инструкцией запроса ключа».

## 6.7 Footer

- контакты поддержки;
- юридический блок;
- ссылка «Для разработчиков» на upstream Goose.
- email для связи и ключей: `bavadim@gmail.com`.

Юридический минимум (P0):
- наименование владельца продукта;
- email для официальной связи;
- ссылка на публичную оферту (`/offer` или якорный блок на этой же странице);
- ссылка на политику обработки данных (короткая версия допустима для MVP);
- пометка, что условия предоставления доступа уточняются в переписке при выставлении счёта.

## 7. Внутренняя продуктовая аналитика (для команды, не публиковать)

Это внутренний раздел для прикидок контента.

Допущения по модели уровня `gpt-5.2-codex`:
- input `~$1.75 / 1M`, output `~$14 / 1M`;
- cost = `input_tokens * 1.75 / 1_000_000 + output_tokens * 14 / 1_000_000`.

Примеры:
- лёгкая задача: `1.5k in + 0.5k out` -> `$0.0096`;
- средняя: `6k in + 2k out` -> `$0.0385`;
- тяжёлая: `15k in + 5k out` -> `$0.0963`.

Внутренняя оценка ёмкости демо:
- лёгкие: ~80–140 запусков;
- средние: ~20–50;
- тяжёлые: ~8–20.

Как использовать в публичном copy:
- «Обычно демо хватает для нескольких реальных задач»;
- «Объём зависит от сложности».

Без $/токенов на сайте.

### 7.1 Процесс получения ключа (операционная схема)

Цель:
- дать пользователю максимально простой путь после демо без отдельного биллинга на первом этапе.

Целевой flow:
1. Пользователь нажимает `Запросить ключ` или пишет на `bavadim@gmail.com`.
2. В письме указывает краткий контекст использования.
3. Команда отвечает шаблонным письмом и отправляет счёт на оплату.
4. Пользователь оплачивает счёт и присылает подтверждение (если требуется).
5. Команда отправляет ключ (в течение 1 рабочего дня после подтверждённой оплаты).
6. Пользователь активирует ключ в приложении.

Статусы заявки (внутренне):
- `new` -> `invoice_sent` -> `paid` -> `key_sent` -> `activated` (если подтвердил активацию).

### 7.2 Минимальная аналитика процесса ключей (внутренняя)

Что фиксировать вручную (таблица/ноушен):
- дата первого письма;
- источник (лендинг / личная рекомендация / иное);
- дата ответа;
- дата отправки счёта;
- дата подтверждения оплаты;
- дата отправки ключа;
- факт активации (да/нет);
- причина отказа/паузы (если есть).

Метрики первой итерации (без сложной веб-аналитики):
- `TTR` (time to first reply): медиана времени первого ответа;
- `TTP` (time to paid): медиана времени от первого письма до оплаты;
- `TTK` (time to key): медиана времени от оплаты до выдачи ключа;
- `Activation rate`: доля выданных ключей, которые активированы;
- `Drop rate`: доля заявок, где счёт не оплачен или ключ не активирован.

Ориентиры для текущего этапа (мягкие):
- первый ответ: в тот же или следующий рабочий день;
- счёт: обычно в день обращения;
- выдача ключа: в течение 1 рабочего дня после подтверждённой оплаты.

Риски и меры:
- риск потери писем -> ежедневная ручная сверка inbox;
- риск задержек ответа -> использовать шаблоны писем;
- риск неполных данных -> в автоответе просить минимальный набор данных.
- риск задержек после оплаты -> отдельный ежедневный контроль статуса `paid`.

## 8. Технические требования

## 8.1 Новая папка проекта (вне documentation)

Создать отдельный проект лендинга:
- `site/landing/`

Рекомендуемая структура:
- `site/landing/package.json`
- `site/landing/docusaurus.config.ts`
- `site/landing/sidebars.js` (минимальный, если нужен docs-раздел)
- `site/landing/src/pages/index.tsx`
- `site/landing/src/pages/index.module.css`
- `site/landing/src/components/*` (по необходимости)
- `site/landing/src/config/downloads.ts`
- `site/landing/static/*` (favicon, og-image, бренд-ассеты)

## 8.2 Выбранный стек

Стек для первой версии:
- Docusaurus 3 (React + TypeScript, статическая генерация).

Почему:
- уже используется в репозитории (ниже риск поддержки);
- даёт статический сайт без backend;
- удобно держать одностраничник в `src/pages/index.tsx`;
- стандартный build/deploy для GitHub Pages.

## 8.3 Ссылки на скачивание актуальной версии из CI/release

Источник для лендинга: assets релиза с тегом `stable` в GitHub Releases.

Требование к CI/release:
- публиковать desktop-архивы с **каноническими именами**:
  - `InsightStream-goose-macos-arm64.zip`
  - `InsightStream-goose-macos-x64.zip`
  - `InsightStream-goose-windows-x64.zip` (если есть)
  - `InsightStream-goose-linux-x64.zip` (если есть)

Публичные ссылки на лендинге должны строиться как:
- `https://github.com/<org>/<repo>/releases/download/stable/<asset_name>`

Плюсы:
- всегда ведёт на актуальную stable-сборку;
- не требует логина в GitHub;
- не привязано к run_id workflow.

## 8.4 Конфиг ссылок

В `site/landing/src/config/downloads.ts` хранить карту платформ -> URL.

Вариант A (предпочтительно):
- URL формируются из env на build этапе:
  - `LANDING_GH_OWNER`
  - `LANDING_GH_REPO`
  - `LANDING_CONTACT_EMAIL` (по умолчанию `bavadim@gmail.com`)

Вариант B:
- жёстко в config, если repo фиксирован.

## 8.5 Публикация на GitHub Pages

Новый workflow:
- `.github/workflows/deploy-landing.yml`

Триггер:
- `push` в `main` при изменениях в:
  - `site/landing/**`
  - `.github/workflows/deploy-landing.yml`

Шаги:
1. checkout
2. setup node
3. `cd site/landing && npm ci`
4. `npm run build`
5. deploy `site/landing/build` в `gh-pages` через `peaceiris/actions-gh-pages`

## 8.6 Правила сосуществования с documentation

Так как лендинг переезжает из `documentation` в `site/landing`:
- `documentation/` не использовать для продуктового лендинга;
- при необходимости оставить docs как отдельный technical site (вне этого ТЗ);
- чтобы не перетирать лендинг, старый docs-deploy workflow должен быть:
  - либо отключён,
  - либо перенастроен деплоить в поддиректорию (например `/upstream-docs/`) с сохранением root лендинга.

Для первой итерации рекомендуется:
- публиковать в root только лендинг.
